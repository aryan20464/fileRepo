2.4.3. Unique X.509 Mapping should be present

applications:1. For Your Applications (Web Applications & APIs)This is the primary area of impact. If you build a web application hosted on WebLogic and secure it using Two-Way SSL (Client Certificate Authentication), WebLogic handles the authentication behind the scenes.When a user hits your application URL, their browser prompts them to select an X.509 certificate.The DefaultIdentityAsserter intercepts this certificate, maps the attribute (e.g., CN or E) to a WebLogic user, and automatically logs them into your application.The Risk here: If your application relies on this mapping, John Smith A could accidentally log into John Smith B's application account.2. For the WebLogic Administration Console LoginBy default, when you access the WebLogic Administration Console (https://hostname:7001/console), you see a standard username and password web form.If you use the standard username/password form: This X.509 unique mapping control does not apply to you. The DefaultIdentityAsserter is not triggered by a standard password login.If you have configured Console Redirection for SSL Authentication: If your administrators explicitly configured the WebLogic Console to
bypass passwords and log admins in using their physical smart cards (like CAC/PIV cards) or browser certificates, then yes, this mapping applies to the Console.The Risk here: A low-privileged user with a similar certificate attribute could inadvertently get logged into the Console with full Admin server privileges.SummaryIf you only log into the WebLogic Console using a username and password, this CIS control is dormant for the console itself. However, security auditors flag it because the underlying WebLogic security framework (the security realm) is always active and vulnerable to exploitation if any application on the server utilizes certificate-based logins.
